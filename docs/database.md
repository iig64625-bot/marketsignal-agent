# SignalPulse Database — ER Diagram & Schema Reference  > Database: SQLite (default) or any SQLAlchemy-compatible URL (Postgres / MySQL).   > Migrations: `migrations/` directory managed by Alembic.  ## ER Diagram  ```mermaid erDiagram     companies ||--o{ sources : "owns"     companies ||--o{ signals : "subject of"     companies ||--o{ reports : "battlecard target"     sources ||--o{ raw_documents : "produces"     raw_documents ||--|| normalized_documents : "cleans into"     normalized_documents ||--o{ document_chunks : "split into"     normalized_documents ||--o{ events : "extracts"     events ||--o{ signals : "supports"     crawl_runs ||--o{ raw_documents : "tracks"     crawl_runs ||--o{ signals : "tracks"     crawl_runs ||--o{ reports : "produces"     crawl_runs ||--o| eval_runs : "evaluated by"     reports ||--o{ claims : "contains"     claims ||--o{ citations : "supported by" ```  ## Tables  ### `companies` The target + competitor companies. - `id` (PK, 12-char hex) — `signalpulse.models.base.new_id()` - `name` (unique) — e.g. `"Dify"`, `"Coze"`, `"FastGPT"` - `website`, `description`  ### `sources` Data sources for each company (RSS, changelog, GitHub, etc.). - `id`, `company_id` (FK), `source_type`, `url`, `enabled`  ### `crawl_runs` One row per pipeline execution. The root of every other row. - `id`, `started_at`, `finished_at`, `status` (`pending` / `running` / `completed` / `failed`) - `triggered_by` (`cli` / `api` / `scheduler` / `sample` / `test`) - `time_window_start`, `time_window_end` - `error_message` (set if status=failed)  ### `raw_documents` The bytes-as-fetched from a source. Idempotent via checksum. - `id`, `crawl_run_id` (FK), `source_id` (FK), `company_id` (FK) - `url`, `checksum` (sha256 hex), `http_status`, `content_type` - `body` (raw HTML / JSON / RSS XML)  ### `normalized_documents` Cleaned + structured version of `raw_documents`. - `id`, `raw_document_id` (FK, 1:1), `company_id` (FK) - `source_type`, `title`, `clean_text`, `language` - `published_at`, `canonical_url` - `content_hash` (used for dedup), `dedup_group` - `category`, `confidence`  ### `document_chunks` Sliding-window chunks of `normalized_documents.clean_text` for RAG. - `id`, `normalized_document_id` (FK), `chunk_index`, `text` - `start_char`, `end_char`  ### `events` Structured events extracted from documents (product_update, pricing_change, etc.). - `id`, `document_id` (FK), `company_id` (FK) - `event_type`, `title`, `summary` - `published_at`, `confidence` (0-1) - `evidence_json` (list of verbatim source spans)  ### `signals` Market signals (the LLM"s interpretation of multiple events). - `id`, `crawl_run_id` (FK), `company_id` (FK) - `signal_type` (`product` / `pricing` / `hiring` / `gtm` / `risk`) - `finding` (factual), `analysis` (interpretive), `recommendation` (actionable) - `confidence` (`high` / `medium` / `low`) - `supporting_event_ids_json`, `supporting_document_ids_json`  ### `reports` Weekly report + per-competitor battlecards. - `id`, `crawl_run_id` (FK), `report_type` (`weekly` / `battlecard`) - `company_id` (FK, NULL for weekly) - `title`, `markdown_path`, `json_path`  ### `claims` Atomic claims extracted from a report, validated for citation support. - `id`, `report_id` (FK) - `claim_text`, `claim_type` (`fact` / `analysis` / `recommendation`) - `confidence` (0-1), `is_supported` (bool) - `supporting_citation_ids_json`  ### `citations` Evidence URLs that back a `claim`. - `id`, `report_id` (FK), `claim_id` (FK) - `document_id` (FK, nullable), `url`, `snippet`  ### `eval_runs` Aggregate eval metrics per crawl run. One row per run. - `id`, `crawl_run_id` (FK) - `citation_coverage`, `unsupported_claim_rate`, `dedup_rate` - `avg_latency_ms`, `token_cost_usd` - `summary_json` (full breakdown including per-node latencies)  ## Migrations  All schema changes go through Alembic:  ```bash # Apply pending migrations alembic upgrade head  # Create a new migration after editing models alembic revision --autogenerate -m "add X to Y"  # Roll back one migration alembic downgrade -1 ```  Migration history lives in `migrations/versions/`. The initial migration is `fce17478b81d_initial_schema.py`.  ## Initialization  For a fresh dev environment, use the helper script:  ```bash # Just create the tables python scripts/init_db.py  # Also seed the sample dataset (recommended for demos) python scripts/init_db.py --with-sample  # Nuclear option: drop everything and recreate python scripts/init_db.py --reset --with-sample ```  The script is idempotent (`create_all` is safe to call multiple times) and respects the `DATABASE_URL` environment variable.  ## Index Strategy  - PKs: implicit clustered indexes on every table - `crawl_runs.started_at` — DESC index for "list recent runs" - `reports.report_type` — index for "filter by weekly / battlecard" - `sources.company_id` — index for "list a company"s sources" - `signals.company_id` — index for "list a company"s signals" - `normalized_documents.content_hash` — index for dedup - `eval_runs.crawl_run_id` — index for "latest eval for a run" - Foreign keys all carry indexes (via `index=True` on the Mapped column)  ## Production Tips  1. **Switch to Postgres** by setting `DATABASE_URL=postgresql+psycopg://user:pwd@host/db` — every model uses generic SQLAlchemy types so no schema change is needed. 2. **Run migrations in CI** before starting the API: `alembic upgrade head`. 3. **Back up the `crawl_runs` table** first — losing it means losing the audit trail of every report. 4. **Tune SQLite** with `PRAGMA journal_mode=WAL` for concurrent reads (already configured in `db/engine.py`).
